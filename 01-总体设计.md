# 一、项目简介

## 1.1 项目背景

目前浏览器实时语音识别（ASR）能力越来越普及，例如：

- ChatGPT Voice
- Claude Voice
- 豆包 AI
- 通义千问
- 腾讯元宝
- Google Gemini

它们的网页端基本采用相同的技术方案：

```
浏览器麦克风
    ↓
Web Audio API
    ↓
PCM采集
    ↓
WebSocket
    ↓
ASR服务
    ↓
实时返回文本
```

虽然浏览器已经提供了 Web Audio API，但开发者仍然需要自行完成：

- 麦克风权限申请
- AudioContext 初始化
- AudioWorklet 采集 PCM
- Float32 转 PCM16
- 音频重采样
- WebSocket 建立连接
- 腾讯云协议实现
- 波形计算
- 音量计算
- 状态管理
- 生命周期管理
- 错误处理

整个流程涉及多个领域（浏览器音频、WebSocket、DSP、ASR 协议），业务开发者接入成本较高。

因此设计 **RealtimeASR SDK**，统一封装浏览器实时语音识别能力。

---

# 二、项目目标

RealtimeASR 定位：

> 一个专注于浏览器实时语音识别（Realtime Speech Recognition）的 TypeScript SDK。

SDK 提供完整的录音链路，但不负责 UI。

---

## V1.0 支持能力

- 浏览器麦克风录音
- AudioWorklet 音频采集
- PCM16 编码
- 自动重采样（统一 16k）
- 实时音量计算
- 实时波形数据
- WebSocket 上传
- 腾讯云 ASR Provider
- 实时字幕（Partial Result）
- 最终字幕（Final Result）
- 生命周期管理
- 状态机
- TypeScript 支持

---

## V1.0 不包含内容

以下内容由业务实现：

- UI
- 波形样式
- 输入框
- 聊天逻辑
- AI 对话
- TTS
- 音频播放

RealtimeASR 仅负责：

```
Mic → PCM → ASR → Text
```

# 三、设计原则

## 3.1 单一职责

每个模块仅负责自己的职责。

例如：

Recorder 不负责上传。

Provider 不负责录音。

Wave 不负责 UI。

---

## 3.2 Provider 解耦

Core 永远不知道当前接的是腾讯还是阿里。

未来新增 Provider 不修改任何 Core 代码。

---

## 3.3 UI 解耦

SDK 不提供任何 UI。

例如：

```
Canvas
SVG
React
Vue
WebGL
```

均由业务自行实现。

SDK 仅输出数据。

---

## 3.4 面向事件

所有数据统一 EventEmitter。

例如：

```
partial

final

volume

wave

error
```

---

## 3.5 高性能

采用现代浏览器方案：

- AudioWorklet
- Binary WebSocket
- PCM16
- TypedArray
- EventEmitter

避免：

- ScriptProcessor
- Base64
- MediaRecorder（实时场景）

## Recorder

Recorder 是整个 SDK 对外入口。

职责：

- 生命周期管理
- 初始化
- 开始录音
- 停止录音
- 销毁
- 事件派发

Recorder 不关心：

- WebSocket
- 腾讯协议
- Provider

---

## AudioEngine

负责：

- 创建 AudioContext
- getUserMedia
- 创建 AudioWorklet
- 创建 AudioNode

统一管理浏览器音频。

---

## AudioWorkletManager

负责：

- 注册 Worklet
- 接收 Float32Array
- 回调 PCM 数据

输出：

```
Float32Array
```

---

## PCMEncoder

负责：

```
Float32

↓

Int16
```

输出：

```
Int16Array
```

---

## Resampler

浏览器通常：

```
48000Hz
```

腾讯要求：

```
16000Hz
```

因此统一：

```
48000

↓

16000
```

SDK 自动完成。

业务无需关心。

---

## FrameBuffer

腾讯建议：

```
200ms
```

FrameBuffer：

负责累计 PCM。

例如：

```
3200 sample

↓

6400 byte

↓

send()
```

---

## VolumeMeter

负责计算实时音量。

算法：

```
RMS
```

输出：

```
0~100
```

供业务绘制：

```
██████
```

---

## EventEmitter

统一事件中心。

例如：

```
start

stop

partial

final

wave

volume

error
```

---

## StateMachine

负责生命周期。

状态：

```
Idle

Initializing

Recording

Stopping

Stopped

Destroyed
```

禁止非法状态切换。

例如：

Recording

↓

再次 start()

直接抛错。

# 八、Provider 设计

Provider 是整个 SDK 最大的扩展点。

所有 Provider 必须实现统一接口。

```ts
interface ASRProvider {

    connect(): Promise<void>

    send(buffer: ArrayBuffer): void

    close(): void

}
```

Core 永远只调用 Provider。

不知道 Provider 内部实现。

# 十四、为什么不用 MediaRecorder？

MediaRecorder 更适合：

- 录音保存
- 上传文件

而不是实时 ASR。

原因：

- 输出 Opus
- 浏览器兼容性差异
- 不方便实时获取 PCM
- 延迟高

RealtimeASR 采用：

```
AudioWorklet

↓

Float32

↓

PCM16

↓

WebSocket
```

这是目前业内主流方案。

---

# 十五、为什么使用 AudioWorklet？

ScriptProcessorNode 已废弃。

AudioWorklet 优势：

- 官方推荐
- 延迟更低
- 音频线程执行
- 不阻塞 UI
- 更稳定

因此 V1 不考虑 ScriptProcessor。

**设计结论：**

RealtimeASR 采用 **Core + Provider** 架构，实现浏览器录音与 ASR 服务完全解耦。Core 负责音频采集、PCM 编码、重采样、缓冲、事件与状态管理；Provider 负责具体 ASR 服务协议（当前为腾讯云）。整个 SDK 不依赖任何 UI 框架，业务层仅通过事件获取实时文本、音量和波形数据，从而保证高性能、可扩展和长期可维护。
